## 0. 移动端的设计原则（在 UX.md §0 之上加 3 条）

7. **触屏不是缩水版鼠标**。move = 滚动、tap = 点击、long-press = hover & right-click 的合体。**不要硬把桌面交互"缩小"塞到手机上**——重新想"在 5 寸玻璃上做这件事最自然的姿势是什么"。
8. **大拇指可达**。所有常用动作（add、切 tab、完成、放弃）都要能在屏幕**下半部分**单手操作。卡顶的 + 子任务、卡顶右侧菜单按钮等不算常用，可以放上半部分。
9. **键盘是软键盘**。文本输入唯一会触发软键盘（Quick Add 唤起）；其余路径都要在没物理键盘的前提下 100% 可走通。所有 `Ctrl+1` `D` `/` 等桌面快捷键移动端**不存在**——必须有等价的视觉 / 手势入口。

## 1. 表面（Surfaces）— 移动端的差异

| Surface | 移动端形态 | vs 桌面 |
| --- | --- | --- |
| **看板视图（Task Board）** | 全屏 leaf；底部 sticky 操作条（垃圾桶 + Quick Add 按钮） | 桌面是 tab 内一个 leaf；移动端通常占满整个屏幕 |
| **状态栏小部件** | **不存在**——Obsidian Mobile 没暴露稳定的状态栏槽位 | 桌面 `📋 N today · ⚠ M overdue` 信息**嵌进 board header**（见 §3.4），不再做独立小部件 |
| **Quick Add** | 底部 sheet（半屏弹起），自动避让软键盘 | 桌面是顶部一行 input |
| **Obsidian CLI 动词** | **完全不存在**——Obsidian Mobile 不支持 native CLI（v1.12.x 仍仅桌面） | 桌面才有；AI agent 路径**仅桌面** |

**移动端 out-of-scope**（与 UX.md §15 一致 + 加这些）：

- 无 CLI（mobile 平台限制）→ AI agent 流程不在移动端运行
- 无 hover popover（无 mouseover 事件可触发）→ 用 long-press 替代
- 无键盘快捷键 → 用手势 / 按钮替代
- 无右键菜单 → long-press 替代
- 无 status bar 小部件 → 信息嵌 board header

> 拟新增故事 `US-501`：**移动端不支持的桌面功能（CLI、hover popover、键盘快捷键）静默 no-op，不弹错误 / 警告**——用户在移动端看不到这些功能存在的痕迹。

## 2. 信息架构（IA）— 移动端布局

```
┌────────────────────────────────────────┐
│  [周][月][已完成][未排期]    🔎  ⚙   │  ← 顶部 tab 条（横向滚动）+ 筛选 / 设置
├────────────────────────────────────────┤
│  📋 3 today · ⚠ 1 overdue              │  ← Header 嵌入 status 信息（替代桌面 status bar）
│                                        │     (US-507 草案)
├────────────────────────────────────────┤
│                                        │
│       [当前 tab 主区]                   │
│                                        │
│       (周/月/已完成/未排期)             │
│                                        │
│                                        │
│                                        │
├────────────────────────────────────────┤
│       未排期池                          │  ← 仅周/月 tab 显示，可向上拖展开
│       [卡] [卡] [卡] ...               │
├────────────────────────────────────────┤
│   🗑              ➕ Add               │  ← Bottom action bar (sticky)
└────────────────────────────────────────┘
```

- **顶部 tab 条**：4 个 tab + 设置入口。手机宽度不足时可横向滚动（不要折成两行）。每 tab 仍带未读小圆点（US-105），但不显示快捷键提示（移动端无键盘）。
- **Header status row**：替代桌面状态栏，永远显示 `📋 N today · ⚠ M overdue`，与桌面 US-106 等价。点击无效（已经在看板内）。
- **未排期池**：默认折叠到屏幕下方，可向上拖手柄展开 / 收起。展开时占屏幕下半部 ~40%。
- **Bottom action bar**：`🗑 垃圾桶`（sticky，拖入 = 放弃，US-123）+ `➕ Add` 按钮（唤起 Quick Add bottom sheet）。两侧可达。

**横竖屏适配**（拟 `US-502`）：
- 竖屏：上述布局
- 横屏（iPad / 大屏手机）：可考虑回到 7 列周视图——以**屏幕宽度 ≥ 600px** 作为分界（CSS `@media (min-width: 600px)`）；< 600px 走移动端布局，≥ 600px 走桌面布局。**判断条件用 CSS 不用 UA**（避免 user-agent 漂移）。

## 3. 各 Tab 的移动端规格

### 3.1 周视图（移动端默认）

桌面 7 列在手机上不可行。两个候选：

- **A. 垂直 list**（推荐）：7 行 = 7 天，每行一个折叠面板，row header = `星期 MM-DD · N tasks · XhYm`，点 row header 展开 / 折叠该日卡片列表。**当前日默认展开**，其余默认折叠。
- B. 滑动切日：单日全屏视图 + 左右滑动切日。优点视觉聚焦、缺点不容易"扫一周哪天满"

**选 A**。理由：
- 垂直 list 滚动一致（不与卡片内的纵向滚动打架）
- "本周整体感"在 row header 的 `N tasks · XhYm` 行就有，对应桌面 US-116
- 拖拽改期目标都是 row header 一行，跨日拖拽自然
- 实现量小（同一份 tasks 数据，按天分组渲染）

**列序**：跟设置（周一为首 / 周日为首，US-112）。共 7 行。
**今日 row**：背景 = `--background-secondary-alt`，row header 加 `今天` 小标签。
**翻周**：顶部增加 `< 今 >` 按钮组（< 上一周 / 今天 / 下一周 >）；不依赖键盘。
**拖入目标**：每行整个 row body 是 drop zone（包括展开后的卡片缝隙）；折叠状态下 row header 也是 drop zone。

> 拟 `US-503`：周视图在移动端默认垂直 list（7 行折叠面板）；当前日默认展开，其余折叠。row header 显 `N tasks · XhYm`。

### 3.2 月视图

保留 6×7 日历，但每格简化：

- 数字（日期）+ 一个**任务数小圆点**（无任务则不显）
- 当日格高亮
- **不显**任何卡片缩略（手机格子太小放不下）
- 点格 = 弹一个该日 list（半屏 bottom sheet 显示该日所有任务）
- 拖入仍是日期格（drop target，US-122）；松手即改 `⏳`

> 拟 `US-504`：月视图在移动端简化为"日历 + 数字 + 任务数圆点"，点格弹该日 list；不在格内画卡片。

### 3.3 已完成视图

桌面已有的"按周分组、本周展开、历史折叠"（US-304）在移动端逻辑不变。**只有视觉变化**：

- 每周组顶 stat（`准确率 / top tag 时长`）从一行变成两行（手机宽度不够）
- 卡片用紧凑模式（见 §4）

### 3.4 未排期视图

桌面 masonry 布局在手机上窄屏单列即可（`column-count: 1`）。其它（按 `groupTags` 分组、顶部"按 deadline 排序"说明文字、无匹配进"其他"）保持 UX.md §4.4 不变。

## 4. 卡片（Card）— 紧凑模式

桌面卡（UX.md §5.1）在手机上要瘦身：

```
┌────────────────────────────────────────┐
│ [✓] 任务标题                  ⏳ 04-25  │  ← 行 1（始终显示）
│ #2象限 · 90m / 75m            📅 05-15  │  ← 行 2（meta + tags 合并到一行，省略号）
│   ├ [ ] 子任务（仅显 1 层）   ⏳ 04-26  │  ← 行 3+（subtask，仅 1 层；超 1 层显 "+N"）
└────────────────────────────────────────┘
```

**与桌面的差异**：
- **meta + tag 合并到一行**（桌面是分两行）。空间不够时按 `tag, est, actual, 📅` 顺序保留，超出截断 + `…`。
- **子任务只显 1 层**（桌面是递归全显，UX.md §5.3 / US-142）；> 1 层时在父行末显 `+N` 提示，点开弹半屏 bottom sheet 显完整树（保持 US-142 的"递归显示"语义，只是默认折叠到 1 层视觉减负）
- **卡间距**减到 6px（桌面 8px），padding 减到 8/10（桌面 12/16）
- **不显 hover popover**——hover 不存在，long-press 替代（见 §5）

**状态机视觉差异**：
- overdue / near-deadline 仍用左侧 3px 红 / 黄条（US-115）— 在窄屏上比 badge 更易扫
- 拖动状态：`opacity 0.85 + shadow --shadow-l` 改成 `opacity 0.9 + scale 1.04 + shadow --shadow-l`（移动端缺少阴影感知，加 `scale` 让用户知道"抓起来了"）

> 拟 `US-505`：移动端卡片紧凑模式——meta 行合并 / 子任务默认显 1 层 / 拖动加 `scale 1.04`。

## 5. 交互 — 触屏专用

### 5.1 Long-press（长按）= 桌面右键 + hover

按住卡片 ≥ 500ms 弹半屏 bottom sheet，内容 = 桌面右键菜单（UX.md §6.5）+ 桌面 hover popover（UX.md §5.4）合并到一个 sheet：

```
┌──────────────────────────────────────┐
│  📍 Tasks/Inbox.md:L42                │  ← 源文件路径（hover popover 信息）
│  父链：项目 X › 阶段 2 › 当前任务      │
│  上下文（源文件 ±2 行）...             │
│  ──                                   │
│  打开源文件                            │
│  切换完成                              │
│  ──                                   │
│  安排到今天 / 明天 / 清空 ⏳            │
│  ──                                   │
│  分组 → <用户配置>                    │
│  ──                                   │
│  放弃                                 │
└──────────────────────────────────────┘
```

- **延时 500ms**（桌面 hover popover 是 700ms；移动端快一点更跟手）
- 外部点 / 下滑关闭
- **绝不**在 long-press 触发时同时启动 drag——必须明确二选一：500ms 内移动手指 = 拖拽起；不动 = long-press menu

> 拟 `US-506`：长按卡片 ≥ 500ms 弹"右键菜单 + 源信息"合并 sheet；按住但移动 ≥ 4px = 改走拖拽（500ms 也可以，先到先得），不要同时触发。

### 5.2 拖拽

- **进入 drag mode**：长按 250ms（不是 500ms）+ 移动 ≥ 4px。比 long-press menu 更早进入 drag，让"想拖就拖"占优先级。
- **手指拖动反馈**：原位置变虚线占位 + 跟随手指的浮卡 `scale 1.04 + shadow --shadow-l`。
- **drop target 高亮**：合法目标 `--background-modifier-hover`；非法目标（自己 / 后代，US-126）`--color-red` + 禁止图标。
- **跨 tab dwell**（拖动时悬停 tab 头自动切）：**800ms**（桌面 600ms，手指比鼠标抖、用户更可能误悬停一会儿；800ms 更稳）。tab 头出现进度条。
- **自动滚动**：手指停在屏幕上下边缘 60px 内（桌面 40px，手机指头粗）按 200px/s 自动滚动。
- **垃圾桶**：始终在 bottom action bar 内 sticky 可见。
- **嵌套到另一卡**：松手在另一卡卡体上 = 跨文件 nest（US-125），等价桌面拖到另一卡，toast + undo（UX.md §6.2）。

> 拟 `US-507`：移动端拖拽 long-press 250ms 进入 drag mode；跨 tab dwell 800ms；自动滚动边缘 60px。

### 5.3 滑动手势（移动端独有）

- **左滑卡片** → 标完成（等价 `Space` 切完成）。手势确认窗口：露出绿底"完成"提示 + 1 秒撤销（toast + undo）。
- **右滑卡片** → 放弃（等价拖到垃圾桶）。手势确认窗口：露出红底"放弃"提示 + 1 秒撤销。
- 滑动距离阈值 ≥ 30% 卡宽，松手回弹 OR 触发；< 30% 取消。
- **不**做 "swipe-to-delete"（语义不一致，会和"放弃"打架）。

> 拟 `US-508`：左滑 = 完成、右滑 = 放弃；阈值 30% 卡宽，1 秒 toast undo。方向选择基于"左滑积极推进"（mail / inbox 通用模式）。

### 5.4 Quick Add（Bottom sheet）

- `➕ Add` 按钮（bottom action bar 右）唤起半屏 sheet，从底部弹起
- sheet 内 = 一行 input（与桌面 UX.md §6.6 同），软键盘自动出现
- **键盘避让**：sheet 高度 = 屏幕剩余高度；软键盘弹出时 sheet 上移到键盘上方 + 8px 间距
- 自然语言日期解析照旧（中英两套，US-410）
- 唯一写入位置仍是当日 daily（US-163）

> 拟 `US-509`：Quick Add 在移动端是 bottom sheet，软键盘弹出时自动避让，sheet 上移到键盘上方 + 8px。

### 5.5 不存在 / 静默替代

| 桌面交互 | 移动端 |
| --- | --- |
| `Ctrl+1~4` 切 tab | 直接点 tab（顶部条） |
| `/` 聚焦筛选 | 顶部 🔎 按钮唤起 search overlay |
| `⌘/Ctrl+T` Quick Add | 底部 ➕ 按钮 |
| `⌘/Ctrl+Z` Undo | toast 内的 "撤销" 按钮（per-action）+ 设置里"最近撤销栈"长按入口 |
| `1/2/3/4` 设组 tag | long-press menu → 分组 |
| `←/→` 改 ⏳ ±1 天 | long-press menu → 安排到 今天/明天/清空 |
| `D` 弹日期对话框 | long-press menu → "选日期"（弹原生 date picker） |
| `Space` 切完成 | 左滑 OR long-press menu → 切换完成 |
| `E/Enter` 打开源 | long-press menu → 打开源文件 |
| `Delete/Backspace` 放弃 | 右滑 OR long-press menu → 放弃 |
| 鼠标悬停 popover | long-press menu 顶部信息区 |
| 鼠标右键菜单 | long-press menu |
| 拖拽自动切 tab dwell 600ms | dwell 800ms |
| 卡 `+ 子任务` 按钮（hover 出现） | 卡上**始终可见**的 `+` 小按钮（hover 在移动端无） |

## 8. 空状态 / 错误态 / 加载态

- **空状态文案不变**（与 UX.md §8.1）；按钮位置改：从屏幕中央按钮改成"空状态文案 + 中央 ➕ 按钮一组"，确保大拇指可达

## 9. i18n & 视觉变量

**全部沿用 UX.md §10 / §11**，无差异。强调一条移动端必须满足的：

- **不要在 i18n 字符串里写桌面特定字面**（如 `Ctrl+T` / `右键菜单`）——这些在移动端没意义；如需在 UI 里说"按 ⌘T 加任务"这类，应**根据平台条件渲染**两套：桌面文案 + 移动端文案，而不是翻译两套快捷键字符串。

> 拟 `US-510`：UI 文案中桌面特定的快捷键 / 鼠标操作描述需要平台条件分支；不要把 `Ctrl+T` 翻译成中文版的 `Ctrl+T`，应在移动端文案里改写成 `点 ➕ 按钮`。

## 10. 性能感知（移动端）

- **打开看板首次** ≤ **2.5s**（移动设备解析 vault 慢，比桌面 1.5s 宽松）
- **拖拽落定 → 渲染** ≤ 150ms（vs 桌面 100ms）
- **滑动手势识别延迟** ≤ 50ms（手势阈值判定要快）
- **未打开看板时插件感觉不存在** —— 同桌面 BUG.md 反向验收，仍是硬指标

- `US-510` UI 文案中桌面特定的快捷键 / 鼠标操作描述要平台条件分支；不在 i18n 字符串里翻译 `Ctrl+T` 这类描述——移动端写 `点 ➕ 按钮`。

## 13. 待 @Tiger / 移动实现者决策的影响项

UX 决策对实现的硬约束（仿 UX.md §16，仅移动端独有）：

1. **CSS 媒体查询而非 UA 检测**用于横竖屏 / 桌面 vs 移动布局分支
2. **触摸事件优先**：用 PointerEvent 统一鼠标 + 触屏 + 笔，而非各自 mouseevent / touchevent 一份逻辑
3. **`prefers-reduced-motion` 在移动端尤其重要**——很多用户开了系统级动效降级，必须遵守（UX.md §11.2）
4. **滑动手势的 30% 阈值 + 1 秒 toast undo** 不能用 setTimeout 漂移；用 rAF 测 elapsed（UX.md §16-8 的延伸）
5. **bottom sheet 软键盘避让**用 `visualViewport.height` 计算，不用窗口高度（避免 iOS 软键盘高度判错）
6. **long-press 500ms vs drag 250ms 是同一手势的不同分支**——必须由同一手势 controller 管理（PointerEvent + 移动距离判定），不能两个独立 listener
7. **plugin 在移动端检测 platform**：`Platform.isMobile` (Obsidian API)，**仅作为 UI 层条件**——业务逻辑和数据层不分平台
